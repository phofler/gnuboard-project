
# 작업 시 유의사항 (Project Guidelines)

## 1. CSS 파일 구조 및 관리 규칙 (Mandatory)
**작업 전 필수 확인: 모든 CSS 작업은 아래 구조에 맞춰 해당 파일에만 수행할 것.**
한 파일(`default.css`)에 모든 코드를 몰아넣는 것을 금지하며, 기능별로 분리하여 관리한다.

### CSS 파일 구성 (4-File Structure)
1.  **`theme/corporate/css/default.css`**
    *   **역할:** 공통 뼈대 및 메인 스타일
    *   **포함:** CSS Reset, 공통 변수(:root), 폰트, 전체 레이아웃 틀, 푸터(Footer), 메인 페이지 전용 스타일
2.  **`theme/corporate/css/menu_top.css`**
    *   **역할:** 상단 헤더 및 메뉴 전용
    *   **포함:** GNB(상단메뉴), 전용 모바일 메뉴, 상단바 디자인
3.  **`theme/corporate/css/sub_common.css`** (구 `sub_img`)
    *   **역할:** 서브 페이지 공통 요소
    *   **포함:** 서브 비주얼(Hero 이미지), 페이지 타이틀, 네비게이션(Breadcrumb), 서브 공통 여백
4.  **`theme/corporate/css/sub_style.css`**
    *   **역할:** 서브 페이지 개별 콘텐츠
    *   **포함:** 회사소개 본문, 표(Table), 폼(Form), 버튼, 기타 콘텐츠 내부 스타일

### 작업 원칙
*   CSS 수정 시 **무조건 전체 파일을 읽지 말고**, 수정하려는 섹션이 포함된 **전용 파일만** 찾아서 작업한다.
*   파일을 합치거나 덮어쓸 때(`write_to_file`)는 **기존 데이터 유실에 각별히 유의**한다. (가능하면 `replace`나 `append` 사용)

## 2. 기존 유의사항 (Legacy)
*   **파일 수정 전 백업 필수:** 중요한 파일 수정 시 반드시 백업본을 확인하거나 생성 후 진행.
*   **그누보드 테마 규칙 준수:** `head.sub.php` 등 코어 파일을 건드릴 때는 테마 로딩 순서를 고려할 것.

## 3. 서브 비주얼(Hero) 작업 규칙 (Critical)
**배경:** 2025-12-17 이슈 해결 (이중 구조 제거 및 이미지 우선순위 정립)

### A. 단일 소스 원칙 (Single Source of Truth)
*   **원칙:** 서브 페이지 상단 비주얼(`Hero Section`)은 **오직 `theme/corporate/head.php` 한 곳에서만 생성**한다.
*   **금지:** 개별 페이지(예: `premium.html`)에서 똑같은 비주얼 영역을 중복해서 만들지 않는다.
*   **구조:** `head.php`가 관리자 설정값을 받아 표준화된 클래스 `section.sub-hero`를 출력하고, CSS는 `sub_common.css`가 담당한다.

### B. 이미지 로딩 우선순위 (Priority Logic)
`head.php`는 다음 순서대로 이미지를 결정해야 한다. (안전장치가 URL을 덮어쓰지 않도록 주의)
1.  **1순위 (Admin File):** 관리자가 업로드한 파일이 있으면 무조건 최우선 사용.
2.  **2순위 (Admin URL):** 파일이 없고 외부 이미지 URL(`Unsplash` 등)이 입력되어 있으면 사용.
3.  **3순위 (Fallback):** 위 두 가지가 모두 없을 때만 테마 기본 이미지 사용.

## 4. 모듈화 가이드라인 (Refactoring Lessons) - 2025.12.17
**시스템의 유지보수성과 확장성을 위해 반드시 지켜야 할 원칙입니다.**

### A. 역할 분리 (Separation of Concerns)
*   **테마(Theme):**
    *   **담당:** 사이트 전체 레이아웃, 공통 네비게이션(Breadcrumb), 공통 디자인 스타일.
    *   **파일:** `head.php`, `tail.php`, `css/sub_common.css`
*   **플러그인(Plugin):**
    *   **담당:** 해당 기능이 보여줘야 할 핵심 본문(Contents) 데이터 출력.
    *   **파일:** `view.php` (순수 데이터 출력만 담당해야 함)
    *   **금지:** 플러그인 파일에서 자바스크립트로 부모 요소(`body`, `#wrapper` 등)의 스타일이나 위치를 강제로 조작하는 행위 금지.

### B. CSS API 패턴 활용
*   플러그인이 테마의 스타일(예: 배경색)을 제어해야 할 경우, 직접 스타일을 덮어쓰지 말고 **규속된 변수 시스템**을 활용하거나, 테마가 제공하는 **클래스 스위치**를 사용해야 한다. (이번 사례에서는 아예 제거하여 테마 기본값인 투명을 활용함)


## 6. 자바스크립트 모듈화 및 위치 규칙 (Script Modularity) - 2025.12.18
**웹 성능 최적화와 안정적인 DOM 제어를 위한 규칙입니다.**

### A. 스크립트 위치 원칙 (Script Placement)
*   **원칙:** 모든 공통 자바스크립트 라이브러리(AOS, Slider 등) 및 초기화 스크립트는 **`tail.sub.php` (페이지 최하단)**에 배치한다.
*   **이유:**
    1.  **성능:** 렌더링 차단 방지 (화면이 먼저 뜨고 스크립트 실행).
    2.  **안정성:** 모든 DOM 요소가 생성된 후 실행되므로 `null` 참조 오류 방지.
*   **금지:** `head.php`나 `head.sub.php`에 무거운 스크립트를 직접 넣는 것을 지양한다. (CSS와 필수 메타태그만 유지)

### B. 중복 제거 (DRY)
*   개별 플러그인(`view.php`)이나 스킨(`content.skin.php`)에서 라이브러리를 중복 로드(`script src=...`)하지 않는다.
*   테마(`tail.sub.php`)에서 한 번만 로드하고, 개별 파일은 순수 HTML/Data 출력에만 집중한다.

## 7. 스킨 개발 및 확장 규칙 (Skin Development Policy) - 2025.12.18
**확장성을 고려한 다중 스킨 관리 원칙입니다.**

### A. 독립형 스킨 구조 (Self-Contained)
*   **원칙:** 새로운 스킨(예: `type_a_1.html`)을 추가할 때는 **HTML 구조와 전용 CSS(`style` 태그)를 하나의 파일에 모두 포함**시킨다.
*   **이유:**
    1.  **확장성:** 스킨이 수십 개로 늘어나더라도 글로벌 CSS 파일(`sub_style.css`)이 비대해지는 것을 방지한다.
    2.  **이식성:** 파일 하나만 복사하면 스킨이 즉시 작동하며, 관리자 에디터(SmartEditor2)에서도 별도 조치 없이 디자인이 바로 적용된다.
*   **예외:** 모든 스킨에서 공통으로 반드시 써야 하는 아주 기본적인 레이아웃(예: `.content-wrapper`) 정도만 글로벌 CSS에 둔다.

## 8. 스킨 CSS 및 캐시 처리 정책 (Specificity & Caching) - 2025.12.18
**프론트엔드 스타일 충돌 및 관리자 캐시 문제를 방지하기 위한 필수 지침입니다.**

### A. CSS 우선순위 해결 (Class Strategy)
*   **문제:** 스킨의 텍스트 색상이 테마의 기본 스타일(예: `body p`)에 밀려 적용되지 않는 경우 발생.
*   **해결:**
    1.  상속(`Inheritance`)에 의존하지 않는다. (예: 부모 div에만 color 지정 금지)
    2.  `!important` 사용을 지양한다.
    3.  **원칙:** **반드시 대상 요소에 전용 클래스를 부여**하여 직접 타겟팅한다. (예: `<p class="intro-text">` → `.intro-text { color: ... }`)

### B. 관리자 스킨 로딩 (Cache Busting)
*   **문제:** 관리자 페이지에서 스킨 로드(`$.ajax`) 시 브라우저가 강력하게 캐싱하여 수정 사항이 반영되지 않음.
*   **해결:** URL 뒤에 타임스탬프(`Date.getTime()`)보다 강력한 **`Math.random()`** 을 파라미터로 붙여 매 요청마다 새로운 파일을 가져오도록 강제한다.

## 9. 플러그인 출력 및 설정 연동 규칙 (View Responsibility) - 2025.12.18
**관리자 설정값이 누락되지 않도록 하기 위한 뷰(View) 파일 작성 원칙입니다.**

### A. 배경색 및 컨테이너 스타일 처리
*   **문제:** 스킨 HTML 파일에만 몰두하다가, DB에 저장된 **배경색(`co_bgcolor`)이나 여백 설정**을 출력하는 코드를 누락하는 실수 발생.
*   **원칙:** **스킨 파일(`type_*.html`)은 내부 콘텐츠** 구조만 담당하고, **뷰 파일(`view.php`)이 전체를 감싸는 래퍼(Wrapper)** 역활을 수행하며 DB 설정을 반영해야 한다.
*   **예시:**
    ```php
    // view.php
    $bg_color = $co_row['co_bgcolor'] ? $co_row['co_bgcolor'] : 'transparent';
    echo '<div style="background-color:'.$bg_color.'">'; // 래퍼에서 배경색 처리
    echo get_skin_content(...); // 스킨 본문 출력
    echo '</div>';
    ```

## 10. 상단 메뉴 CSS 관리 규칙 (Top Menu Strategy) - 2025.12.18
**이중 구조(Dual-Track) 전략: 기본 테마와 확장 플러그인 스타일의 공존 원칙**

1.  **`theme/.../menu_top.css` (테마 내장)**
    *   **역할:** 플러그인 미사용 시 작동하는 **순정 GNB 스타일**.
    *   **규칙:** 이곳의 코드는 플러그인 스킨 디자인에 간섭하지 않도록 최소한으로 유지한다.

2.  **`plugin/.../skins/스킨명/style.css` (플러그인 모드)**
    *   **역할:** 특정 디자인(Modern, Dark 등)을 구현하는 **독립 스타일**.
    *   **규칙:**
        *   기존 CSS 파일을 건드리지 않고, 해당 스킨 파일(`style.css`) 내에서만 작업한다.
        *   **Prefix 필수:** `.gnb_list` 같은 흔한 이름 대신 `.modern-gnb`, `.dark-menu` 같은 **고유 접두어**를 사용하여 테마 CSS와의 충돌을 원천 차단한다.

## 11. 관리자 토큰/세션 오류 해결 가이드 (Troubleshooting) - 2025.12.19

*   **증상:** 게시판 설정 저장, 스킨 적용 등 폼 전송 시 `ajax.token.php` 오류나 `KeyMismatch` 경고창이 뜨며 동작하지 않음.

## 12. 상단 메뉴 스킨 작업 시 필수 유의사항 (Top Menu Skin Precautions) - 2025.12.19
**테마 레이아웃과 플러그인 스킨 간의 충돌 방지를 위한 핵심 기술 지침입니다.**

### A. 테마 래퍼(Wrapper) 충돌 해결
*   **증상:** 상단 메뉴 스킨(Mega Menu 등)을 넓게(예: 1600px) 만들어도, 테마 헤더의 컨테이너(`div#hd_wrapper`)가 **1400px** 등으로 제한되어 있으면 메뉴가 찌그러지거나 정렬이 틀어짐.
*   **해결:**
    1.  **전체 너비가 필요한 스킨**의 경우, 테마 파일(`head.php`)에서 **`div#hd_wrapper` 태그를 주석 처리하거나 제거**해야 한다.
    2.  스킨 파일(`menu.skin.php`) 내부에서 **직접 `div.gnb_wrap` 등을 최상위**에 배치하여 레이아웃 너비를 제어한다.
    3.  CSS 오버라이드(`!important`) 방식보다 **HTML 구조 자체를 상위(Top Level)로 올리는 것**이 가장 확실한 해결책이다.

### B. 샘플 디자인 이식 원칙
*   사용자가 제공한 샘플 파일(`menu_mega.html`)이 있을 경우, 기존 스킨 CSS에 끼워 맞추려 하지 말고 **샘플 파일의 HTML 구조와 CSS를 그대로 복제하여 이식**하는 것이 가장 정확하다.

## 14. 관리자 레이어 팝업 최상위 노출 (Layer Modal Stacking Fix) - 2025.12.20
**복잡한 관리자 레이아웃(Sidebar/Header)에 의해 팝업이 가려지는 문제 해결.**

### A. 문제 상황
*   **증상:** z-index를 99999로 설정했음에도 불구하고, 레이어 팝업이 사이드바나 상단 메뉴 아래에 깔림.
*   **원인:** 팝업 HTML 코드가 특정 div 컨테이너(Stacking Context) 내부에 위치하여, 부모 요소의 z-index나 overflow:hidden 속성에 갇힘.

    `javascript
    .ready(function() {
        body.append(#previewModal); // 팝업을 body 로 이동
        body.append(#unsplashModal);
    });
    `
*   **결과:** 팝업이 DOM 트리의 최상위에 위치하게 되어, position:fixed가 Viewport 기준으로 정상 작동하며 모든 방해 요소를 덮을 수 있음.

## 19. 스킨 연동 구현 시 필수 유의사항 (Skin Integration Logic) - 2025.12.30
**기존 스킨 UI를 타 플러그인(Pro Menu 등)에서 재구현할 때의 스타일링 원칙입니다.**

### A. 문제 상황 (Problem)
*   **증상:** 기존 스킨(`top_menu_manager/skins/basic`)의 HTML 구조를 그대로 가져와서 출력했음에도, 글자 말줄임(`text-overflow`)이나 서브 메뉴 숨김(`display: none`) 처리가 동작하지 않아 레이아웃이 깨짐.
*   **원인:** 해당 기능들을 제어하는 **CSS 파일(`style.css`)이 로드되지 않았기 때문**. HTML 태그와 클래스명만 맞춘다고 스타일이 적용되는 것이 아님.

### B. 해결 원칙 (Solution)
*   **CSS 재작성 금지:** 스킨이 이미 가지고 있는 기능을 위해 `inline style`이나 별도 CSS를 새로 짜지 않는다.
*   **CSS 파일 로드:** 해당 스킨의 `style.css` 파일을 **`add_stylesheet` 함수를 통해 명시적으로 로드**해야 한다.
*   **코드 예시:**
    ```php
    // 설정된 스킨명 가져오기
    $skin_path = G5_PLUGIN_PATH . '/top_menu_manager/skins/' . $top_menu_skin;
    $skin_url = G5_PLUGIN_URL . '/top_menu_manager/skins/' . $top_menu_skin;
    
    // 스킨 CSS 로드 (핵심)
    if (file_exists($skin_path . '/style.css')) {
        add_stylesheet('<link rel="stylesheet" href="' . $skin_url . '/style.css">', 0);
    }
    ```

## 20. 데이터 소스 변경 시 스킨 유지보수 원칙 (Backend Adapter Pattern) - 2025.12.30
**데이터 출처나 로직이 변경될 때, 스킨(Frontend)을 훼손하지 않고 연동하는 핵심 원칙입니다.**

### A. 실패 사례 (Failure Case)
*   **상황:** 메뉴 데이터를 기존 1개 테이블에서 **2개 테이블(Pro Menu + Tree Category) 병합 방식**으로 변경함.
*   **실수:** 병합된 데이터 구조가 기존 스킨과 맞지 않자, **스킨의 CSS와 HTML 구조를 억지로 수정**하여 끼워 맞추려고 시도함.
*   **결과:** '스킨 교체 시 디자인이 자동 변경된다'는 핵심 기능이 파괴됨. 데이터가 바뀔 때마다 모든 스킨을 일일이 수정해야 하는 유지보수 지옥 발생.

### B. 해결 원칙 (Solution Principle)
1.  **백엔드 어댑터 (Backend Responsibility):**
    *   데이터 로직(`lib.php` 등)이 '통역사' 역할을 해야 한다.
    *   서로 다른 테이블에서 가져온 날것의 데이터(Raw Data)를 가공하여, **스킨이 원래 기대하던 '표준 데이터 포맷'으로 완벽하게 변환**한 후 넘겨줘야 한다.
2.  **프론트엔드 격리 (Frontend Isolation):**
    *   스킨 파일(`view.php`, `style.css`)은 데이터가 어디서 왔는지 알 필요가 없어야 한다.
    *   **절대 원칙:** 데이터 구조가 안 맞는다고 해서 **잘 돌아가던 스킨의 CSS나 HTML을 뜯어고치지 않는다.**

### C. OS 경로 호환성 (Cross-Platform Path)
*   **문제:** 로고나 CSS 파일 경로를 리눅스 스타일(`/`)로 하드코딩하여, 윈도우 로컬 개발환경(`\`)에서 엑박(404) 발생.
*   **해결:** 파일 시스템 경로에는 `G5_DATA_PATH`, 웹 URL에는 `G5_DATA_URL`이나 `G5_PLUGIN_URL` 상수를 사용하여 **OS에 상관없이 절대 경로가 자동 생성**되도록 처리한다. "내 컴퓨터에서만 되는 코드"를 방지하는 필수 습관.
